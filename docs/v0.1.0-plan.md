# ORC Release Plan — v0.1.0

Target version: `v0.1.0` (local-first, stable, multi-agent ready)

> Cross-agent collaboration, session continuity, scheduled jobs, and a properly tested system.
> This is the release where ORC stops being "works on my machine" and becomes genuinely useful across a full agent workflow.

---

## What Shipped in v0.0.1

- SQLite core: tasks, memories, jobs, sessions, prompts, projects — all auto-created via `setupDb()`
- REST API + SDK + CLI — all commands working end-to-end
- MCP server with memory, task, job, context, and session tools
- Job executor: `Bun.spawn()`, streaming logs, exit codes
- Scheduler code (cron/repeat/watch) — exists, not activated
- Hook scripts: `post-tool-use.ts`, `pre-compact.ts`, `session-start.ts` for Claude Code only
- 37/37 automated tests passing

**Also shipped since v0.0.1:**

- Dual FTS5 search: Porter + Trigram tables with 3-layer cascade (porter AND → OR → trigram AND → OR → LIKE)
- Memory types: `fact | decision | event | rule | discovery` with `title` field
- Priority-tiered session snapshot (P1/P2/P3 with budget trimming to ≤2KB)
- Session event dedup (SHA256 hash, last-5 window) + FIFO eviction at 1000 events
- Importance + type-weighted `context` scoring
- `memory_delete` tool
- `data_hash` column on `session_events` with safe ALTER migration
- `orc daemon` command: start/stop/status with PID file management, SIGTERM shutdown
- `orc home` command: shows `~/.orc` directory, daemon state, active config
- `orc session list|show` CLI + `GET /sessions`, `GET /sessions/:id` API routes
- Hook configs: `hooks/claude-code/settings.json`, `hooks/codex/settings.json`, `hooks/cursor/mcp.json`
- `session_event` MCP tool accepts explicit `session_id` arg (consistent with snapshot/restore)
- `post-tool-use.ts` handles Codex tool names; passes `session_id` correctly as top-level arg
- Pre-compact hook fixed: stores-only, no double inject_context
- Global CLI flags: `--db`, `--port`, `--host`, `--secret`, `--log-level`, etc. override env/config
- All config overridable via env vars (`ORC_DB_PATH`, `ORC_API_PORT`, `ORC_API_HOST`, etc.)
- 47/47 automated tests passing (was 37)

## Session Continuity: Correct Hook Flow

Current behavior has a subtle bug: `pre-compact.ts` writes `inject_context` to stdout which attempts to inject the snapshot before compaction. This is redundant with the `session-start.ts` post-compact restore, and may cause double-injection.

**Correct flow for v0.1.0:**

```
PostToolUse → session_event(...)     # record what happened (fire and forget)

PreCompact  → session_snapshot(...)  # build+store XML, do NOT write inject_context
                                     # the snapshot is just persisted to DB

SessionStart(source: "compact")      # inject_context ← session_restore()
                                     # agent gets the snapshot after compaction

SessionStart(source: "startup")      # inject_context ← context()
                                     # fresh session gets compact task+memory index
```

This ensures the snapshot is only injected once (post-compact), not twice.
